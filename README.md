# tix

Multi-agent coding orchestrator. Each issue runs through a state machine — plan → review → code → review → validate → human-validate → PR — with a fresh `claude -p` agent per state, isolated in its own git worktree, postgres clone, and tmux session. SQLite is the source of truth.

Project-agnostic. Point it at any git repo via `~/.config/tix/config.json` (or `<repo>/.tix/config.json` for shared team values).

## Quickstart

```sh
# 1. Install
pnpm install
pnpm build
ln -sf "$PWD/dist/cli/index.js" ~/.local/bin/tix
chmod +x dist/cli/index.js

# 2. Tell tix where your project lives
mkdir -p ~/.config/tix
cat > ~/.config/tix/config.json <<EOF
{
  "repoPath": "/abs/path/to/your/repo",
  "worktreesRoot": "/abs/path/to/worktrees",
  "agentCommand": "claude"
}
EOF

# 3. (Optional) Postgres template for per-issue DB clones
psql -c 'CREATE DATABASE tix_template TEMPLATE your_dev_db;'
psql -c 'ALTER DATABASE tix_template IS_TEMPLATE true;'

# 4. Scaffold the project's .tix/ folder
cd /abs/path/to/your/repo
tix init                  # creates .tix/config.json + .tix/README.md
tix init --with-prompts   # also copies tix's baseline prompts so you can override them
```

See `src/config.ts` for all config fields. `tix --help` lists every command.

## How it works

- **Issue** = a row in SQLite with a status, body, round counters, and a slot.
- **State machine**: `new → ready → plan → plan_review → code → code_review → validate → awaiting_human → awaiting_pr → done`. Loops via `plan_revise` and `code_revise` capped at 3 rounds.
- **Slot** = a numbered execution lane (1..N). Each slot owns a tmux session, a postgres DB clone, and predictable port offsets.
- **Worktree** = `<worktreesRoot>/tix-issue-<id>` on branch `agent/issue-<id>`, cut from `baseBranch`.
- **Each state spawns a fresh `claude -p` session** with no memory of prior sessions. Author and critic are intentionally different processes — read artifacts cold, no rationalisation.
- **Artifacts** live under `<worktree>/.tix/`: `current.md` (the agent's brief), `plan.md`, `plan-critique.md`, `code-review.md`, `validation/test-plan.md`, `validation/test-results.md`, `agent.log`. All gitignored except project overrides.

## Typical happy path

```sh
# Create the issue (opens $EDITOR, or use --title/--body)
tix new --human-validate --title "fix: foo bar"

# Mark it ready and claim a slot — provisions worktree + postgres clone + tmux session
tix status 1 ready
tix claim 1

# Launch the planner (auto-respawns through every state)
tix spawn 1

# Watch live tool-by-tool output in another terminal
tix logs 1 -f

# Issue progresses automatically: plan → plan_review → code → code_review → validate → awaiting_human

# When it parks at awaiting_human, review the human test plan
tix show-test-plan 1
$EDITOR ~/source/dac-worktrees/tix-issue-1/.tix/validation/test-results.md
tix human-result 1 pass        # or: fail (kicks back) | needs-discussion (pauses)

# Publisher commits + squashes + pushes the branch and prints the PR-create URL
tix logs 1 -f                  # follow the publisher
# (open the PR in your git host, merge when ready)

# Close out
tix status 1 done --note "PR merged: <url>"
tix release 1                  # tears down worktree, drops the postgres clone, kills tmux, frees slot
```

If an agent calls `tix needs-input <id> "<question>"` mid-run, the issue parks at `needs_input`. Resume with:

```sh
tix resume 1 "your answer here"   # restores prior state, auto-respawns the agent
```

## Useful commands

```sh
tix list                          # all issues, status, slot
tix list --status awaiting_human  # filter
tix slots                         # slot occupancy + port map
tix show <id>                     # full issue + recent events
tix timeline <id>                 # full event history
tix attach <id>                   # tmux switch/attach to the slot's session
tix logs <id> -f                  # tail the agent's tee'd stream live
```

## Project-side customisation

Anything tix needs from a project lives under `<repo>/.tix/`:

- `config.json` — slotCount, baseBranch, postgres template, etc. User-level config still wins on conflict.
- `prompts/<state>.md` — override any baseline prompt for this project. Missing files fall back to tix's defaults.
- `README.md` — autogenerated explainer for future contributors.

Don't want tix in the project anymore? `rm -rf .tix/` and remove the managed block from `.git/info/exclude`. Done.

## Status

- **Phase 1 done**: single-issue end-to-end with auto-respawn between states.
- **Phase 2 in progress**: project overrides, `tix init`, `tix resume`. PR/MR API integration still TODO.
- **Phase 3 deferred**: real daemon (poll-driven respawn instead of timer), per-state timeout enforcement, multi-issue stress testing.

## Tests

```sh
pnpm test            # ~70 tests, ~200ms
pnpm test:watch      # rerun on change
```

Tests focus on the load-bearing logic: state machine transitions, round-cap arithmetic, config loading, format-stream parser, init scaffolding. Anything that shells out (tmux, postgres, git) is verified by smoke runs, not unit tests.
